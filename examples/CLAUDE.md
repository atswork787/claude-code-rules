## 重要なルール

### 1. コード構成

- 少数の大きなファイルよりも多数の小さなファイル
- 高凝集、低結合
- 1ファイルのコード行数(空行・コメント除く)の目安:
  - コンポーネント / フック: 最大200行
  - ビジネスロジック / API: 最大300行
  - 型定義 / 設定ファイル: 最大500行
- 上限を超える場合は責務の分割を検討すること
- 型ではなく機能/ドメインで整理

### 2. コードスタイル

- コード、コメント、またはドキュメントに絵文字なし
- 常にイミュータビリティ - オブジェクトや配列を決してミューテートしない
- 本番コードにconsole.logなし
- try/catchで適切なエラー処理
- Zod等のランタイムスキーマバリデーションライブラリで入力検証

### 3. テスト

- TDD: まずテストを書く
- 最低80%の行カバレッジ(line coverage)
- ユーティリティのユニットテスト
- APIの統合テスト
- 重要なフローのE2Eテスト

### 4. セキュリティ

- ハードコードされた秘密情報なし
- 機密データは環境変数
- すべてのユーザー入力を検証
- パラメータ化されたクエリのみ
- CSRF保護を有効化

## 主要パターン

### APIレスポンス形式

```typescript
interface ApiResponse<T> {
  success: boolean
  data?: T
  error?: string
}
```

### エラー処理

```typescript
try {
  const result = await operation()
  return { success: true, data: result }
} catch (error) {
  logger.error('操作が失敗しました', { error })
  return { success: false, error: 'ユーザーフレンドリーなメッセージ' }
}
```

## 成功指標

- すべてのテストが合格（行カバレッジ80%以上）
- セキュリティ脆弱性なし（npm audit / eslint-plugin-security で警告ゼロ）
- ESLint / Prettier の警告・エラーがゼロ
- ユーザーの要件を満たしている

## 依存関係管理

- パッケージマネージャー: [npm/pnpm/yarn から選択]
- lockファイルは必ずコミットする
- 依存追加時はセキュリティ監査を実施(`npm audit`)

## 禁止事項

- `any`型の使用（やむを得ない場合は`unknown`を使用）
- `@ts-ignore` / `@ts-expect-error`（型エラーは根本解決する）
- `!`(非nullアサーション)の安易な使用

## レビュー優先度

1. セキュリティ
2. 正確性（バグがないこと）
3. パフォーマンス
4. 可読性・保守性

## 情報の参照方針

- 実装や設計で不明点がある場合、推測で進めず以下を参照すること:
  - プロジェクト内のドキュメント(README, docs/配下)
  - MCP経由で利用可能な社内ナレッジベース
  - WebSearchツールによる公式ドキュメントの確認
- ライブラリのAPI仕様は、学習済み知識よりも公式ドキュメントを優先すること

## 検証と報告

- 実装完了時に以下の検証を実行し、結果をユーザーに報告すること:
  - テスト実行結果（合格数 / 失敗数 / カバレッジ）
  - ESLint / Prettier の実行結果
  - npm audit の実行結果
- 検証で問題が見つかった場合は、修正してから再検証すること
- すべての検証が合格するまで「完了」としないこと

## Gitワークフロー

- 慣例的なコミット: `feat:`、`fix:`、`refactor:`、`docs:`、`test:`
- ブランチ名: `feature/<内容>`, `fix/<内容>`, `refactor/<内容>`
- mainに直接コミットしない
- PRはレビューが必要
- マージ前にすべてのテストが合格する必要がある
