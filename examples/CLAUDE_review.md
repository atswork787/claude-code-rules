## B. 追加提案

### B1. 依存関係管理セクション追加

```markdown
## 依存関係管理
- パッケージマネージャー: [npm/pnpm/yarn から選択]
- lockファイルは必ずコミットする
- 依存追加時はセキュリティ監査を実施(`npm audit`)
```

### B2. ブランチ命名規則をGitワークフローセクションに追加

```markdown
- ブランチ名: `feature/<内容>`, `fix/<内容>`, `refactor/<内容>`
```

### B3. 禁止事項セクション追加

```markdown
## 禁止事項
- `any`型の使用（やむを得ない場合は`unknown`を使用）
- `@ts-ignore` / `@ts-expect-error`（型エラーは根本解決する）
- `!`(非nullアサーション)の安易な使用
```

### B4. レビュー優先度セクション追加

```markdown
## レビュー優先度
1. セキュリティ
2. 正確性（バグがないこと）
3. パフォーマンス
4. 可読性・保守性
```

### B5. 情報の参照方針セクション追加

```markdown
## 情報の参照方針
- 実装や設計で不明点がある場合、推測で進めず以下を参照すること:
  - プロジェクト内のドキュメント(README, docs/配下)
  - MCP経由で利用可能な社内ナレッジベース
  - WebSearchツールによる公式ドキュメントの確認
- ライブラリのAPI仕様は、学習済み知識よりも公式ドキュメントを優先すること
```

### B6. 成功指標の具体化

現状(130-137行)の抽象的な表現を具体的な検証基準に変更:

```markdown
## 成功指標
- すべてのテストが合格（行カバレッジ80%以上）
- セキュリティ脆弱性なし（npm audit / eslint-plugin-security で警告ゼロ）
- ESLint / Prettier の警告・エラーがゼロ
- ユーザーの要件を満たしている
```

### B7. 検証と報告セクション追加

```markdown
## 検証と報告
- 実装完了時に以下の検証を実行し、結果をユーザーに報告すること:
  - テスト実行結果（合格数 / 失敗数 / カバレッジ）
  - ESLint / Prettier の実行結果
  - npm audit の実行結果
- 検証で問題が見つかった場合は、修正してから再検証すること
- すべての検証が合格するまで「完了」としないこと
```
