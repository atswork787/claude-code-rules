# コーディングスタイル

本ドキュメントの原則は言語非依存で適用する。
言語固有のルール（型安全性、バリデーションライブラリ、エラー処理パターン等）は `coding-style-{language}.md` を参照すること。

## イミュータビリティ（クリティカル）

理由: 副作用による予期しないバグを防ぎ、変更の追跡とデバッグを容易にする
ルール: 常に新しいオブジェクトを作成し、決してミューテートしない

- オブジェクト: コピーを作成してから値を変更する
- 配列: 破壊的メソッド（push, splice等）ではなく非破壊的な操作（spread, filter, map等）を使用する

## ファイル構成

理由: 変更の影響範囲を局所化し、コードレビューやテストを容易にする
ルール: 小さく焦点を絞ったファイルに分割する（多数の小さなファイルを優先する）
- 高凝集、低結合
- 通常200-400行、最大800行
- 大きなコンポーネントからユーティリティを抽出
- 型ではなく機能/ドメインで整理

## 命名規則

理由: 一貫した命名によりコードの可読性と検索性を高める
ルール: プロジェクト内で命名規則を統一し、名前から役割が推測できるようにする

- 略語を避け、意味が明確な名前を付ける
- boolean変数には is/has/can/should プレフィックスを付ける
- 関数名は動詞で始める（get, create, update, delete, validate等）
- 定数は大文字スネークケース（UPPER_SNAKE_CASE）で統一する

## import順序

理由: import文の順序を統一することで依存関係の把握を容易にする
ルール: 外部依存から内部依存の順に並べ、グループ間は空行で区切る

1. 外部ライブラリ
2. 内部モジュール（エイリアスパス）
3. 相対パス（親から子の順）

## エラー処理

理由: 障害発生時の原因特定を迅速にし、ユーザーに適切なフィードバックを返す
ルール: 予期しない例外を握りつぶさず明示的に処理する

- エラーの原因をログに記録する
- ユーザー向けには具体的で分かりやすいメッセージを返す
- 内部エラーの詳細を外部に漏らさない

## 入力検証

理由: 不正な入力による実行時エラーやセキュリティ脆弱性（インジェクション等）を防ぐ
ルール: システム境界でスキーマバリデーションライブラリを用いてユーザー入力を検証する

## コード品質チェックリスト

作業完了前に:
- [ ] コードが読みやすく適切に命名されている
- [ ] 関数が小さい（<50行）
- [ ] ファイルが焦点を絞っている（<800行）
- [ ] ネストは4レベル以内に抑えている
- [ ] 適切なエラー処理
- [ ] デバッグ用ログ出力を残していない
- [ ] マジックナンバーや環境依存の文字列を定数または環境変数に抽出している
- [ ] ミューテーションがない（イミュータブルパターン使用）
