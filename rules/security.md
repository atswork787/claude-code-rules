# セキュリティガイドライン

## 必須のセキュリティチェック

理由: セキュリティ脆弱性は本番環境への影響が大きく、事後修正のコストが高い
ルール: すべてのコミット前に以下のチェックリストを確認する

- [ ] ハードコードされた秘密情報なし（APIキー、パスワード、トークン）
- [ ] すべてのユーザー入力がスキーマバリデーションで検証されている（`coding-style.md` の入力検証セクション参照）
- [ ] SQLインジェクション防止（パラメータ化されたクエリ）
- [ ] XSS防止（サニタイズされたHTML）
- [ ] Cookie認証を使用する場合、CSRF保護が有効
- [ ] 認証/認可が検証されている
- [ ] APIエンドポイントがある場合、レート制限を設定
- [ ] エラーメッセージが機密データを漏洩しない（`coding-style.md` のエラー処理セクションも参照）
- [ ] 依存パッケージに重大な脆弱性がない（`npm audit` で確認）

## 秘密情報管理

理由: 秘密情報の漏洩はシステム全体の侵害につながり、影響範囲の特定と対処に多大なコストがかかる
ルール: 秘密情報は環境変数で管理し、コードやログに含めない

- 環境変数から秘密情報を取得する（ハードコードしない）
- 秘密情報をログに出力しない（デバッグ時もマスクする）
- `.env` ファイルを `.gitignore` に含める

```typescript
// 決してしない: ハードコードされた秘密情報
const apiKey = "sk-xxxxx"

// 常に: 環境変数
const apiKey = process.env.API_KEY

if (!apiKey) {
  throw new Error('API_KEYが設定されていません')
}
```

## セキュリティ対応プロトコル

セキュリティ問題が見つかった場合:

1. 即座に停止
2. **security-reviewer**エージェントを使用（DB/RLS関連は**database-reviewer**と併用推奨。エージェントが利用できない場合は手動でセキュリティチェックリストを実施）
3. 続行前にクリティカルな問題を修正
4. 露出した秘密情報をローテーション
5. 類似の問題がないかコードベース全体をレビュー
