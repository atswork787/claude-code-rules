# /learn - 再利用可能なパターンを抽出

現在のセッションを分析し、スキルとして保存する価値のあるパターンを抽出します。

## トリガー

以下の条件を1つ以上満たす場合に、セッション中のどの時点でも `/learn` を実行します:

- 複数回の試行・調査を経て解決した問題
- ライブラリやAPIの非公式な動作・制限を発見した場合
- 複数のツールを組み合わせた非自明なデバッグ手順を確立した場合

## 抽出基準

### 対象とするもの

1. **エラー解決パターン**
   - どのようなエラーが発生したか？
   - 根本原因は何だったか？
   - 何がそれを修正したか？
   - これは類似のエラーに再利用可能か？

2. **デバッグ技術**
   - 自明ではないデバッグステップ
   - 機能したツールの組み合わせ
   - 診断パターン

3. **回避策**
   - ライブラリの癖
   - APIの制限
   - バージョン固有の修正

4. **プロジェクト固有のパターン**
   - 発見されたコードベース規約
   - 行われたアーキテクチャの決定
   - 統合パターン
   - プロジェクト固有パターンは **コンテキスト** フィールドに以下を明記すること:
     `**コンテキスト:** プロジェクト固有（[プロジェクト名]）/ 汎用`

### 対象としないもの

- 些細な修正（タイポ、単純な構文エラー）
- 1回限りの問題（特定のAPI障害など）
- 将来のセッションで時間を節約しないパターン

## 出力形式

`~/.claude/skills/learned/[pattern-name].md`にスキルファイルを作成:
※ Windows 環境では `%USERPROFILE%\.claude\skills\learned\` に相当します

ファイル名は kebab-case で命名すること（例: `react-query-cache-invalidation.md`、`nextjs-api-timeout-workaround.md`）

```markdown
# [説明的なパターン名]

**抽出日:** [日付]
**コンテキスト:** [これがいつ適用されるかの簡単な説明] / プロジェクト固有（[プロジェクト名]）または 汎用

## 問題
[これが解決する問題 - 具体的に]

## 解決策
[パターン/技術/回避策]

## 例（任意: コードやコマンドで具体化できる場合のみ記載）

## 使用する場合
[トリガー条件 - このスキルを有効化すべきもの]
```

## プロセス

1. セッションで抽出可能なパターンをレビュー
2. `~/.claude/skills/learned/` の既存スキルと重複・類似がないか確認
   ※ Windows 環境では `%USERPROFILE%\.claude\skills\learned\`
3. 最も価値がある/再利用可能な洞察を特定
4. スキルを集中させる - パターンごとに1つのスキル
5. スキルファイルを下書き
6. スキルファイルの下書きをユーザーに提示し、以下を確認する:
   - ファイル名（`[pattern-name].md`）が適切か
   - 問題・解決策の記述が正確か
   - 保存先でよいか
7. `~/.claude/skills/learned/` に保存
