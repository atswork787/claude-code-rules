# Planコマンド

このコマンドは、コードを書く前に包括的な実装計画を作成するために**planner**エージェントを呼び出します。

## このコマンドが行うこと

1. **要件の再記述** - 構築する必要があるものを明確化
2. **リスクの特定** - 潜在的な問題とブロッカーを表面化
3. **ステップ計画の作成** - 実装をフェーズに分解
4. **計画をファイルに保存** - `plans/plans.md` に計画を書き出す
5. **確認を待つ** - 進める前にユーザーの承認を受けなければならない

## 使用する場合

以下の場合に`/plan`を使用:
- 新機能を開始する
- 重要なアーキテクチャ変更を行う
- 複雑なリファクタリングに取り組む
- 複数のファイル/コンポーネントが影響を受ける
- 要件が不明確または曖昧

## 動作方法

plannerエージェントは以下を行います:

1. **リクエストを分析**し、要件を明確な用語で再記述
2. **具体的で実行可能なステップにフェーズ分解**する
3. **コンポーネント間の依存関係を特定**する
4. **リスクを評価**し、潜在的なブロッカーを特定
5. **複雑さを推定**（高/中/低）
6. **計画を提示**し、明示的な確認を待つ

## 使用例

```
ユーザー: /plan マーケットが解決したときにリアルタイム通知を追加する必要があります

エージェント (planner):
# 実装計画: リアルタイムマーケット解決通知

## 概要
マーケットが解決したとき、監視中のユーザーへリアルタイム通知を送信する機能を追加する。
通知はアプリ内・メール・webhookの複数チャネルをサポートし、確実な配信を保証する。

## 要件
- ユーザーが監視しているマーケットが解決したときに通知を送信
- 複数の通知チャネルをサポート（アプリ内、メール、webhook）
- 通知が確実に配信されることを保証
- マーケットの結果とユーザーのポジション結果を含める

## アーキテクチャ変更
- 新規: lib/notifications.ts（通知サービス）
- 新規: supabase/migrations/add_notifications.sql（通知テーブル）
- 変更: lib/markets.ts（マーケット解決時のフック追加）
- 新規: components/NotificationBell.tsx（フロントエンドコンポーネント）

## 実装ステップ

### フェーズ1: データベーススキーマ
1. **通知テーブルの追加** (ファイル: supabase/migrations/add_notifications.sql)
   - アクション: notificationsテーブルとuser_notification_preferencesテーブルを作成
   - なぜ: 通知の永続化とユーザー設定の管理に必要
   - 依存関係: なし
   - リスク: 低

### フェーズ2: 通知サービス
2. **通知サービスの実装** (ファイル: lib/notifications.ts)
   - アクション: BullMQ/Redisを使用した通知キューと再試行ロジックを実装
   - なぜ: 配信失敗時のリカバリーと非同期処理のために必要
   - 依存関係: フェーズ1完了
   - リスク: 中

### フェーズ3: マーケット解決への統合
3. **解決フックの追加** (ファイル: lib/markets.ts)
   - アクション: ステータスが「解決済み」に変更された際に通知をエンキュー
   - なぜ: マーケット解決イベントと通知送信を連携させるため
   - 依存関係: フェーズ2完了
   - リスク: 中

### フェーズ4: フロントエンドコンポーネント
4. **通知UIの実装** (ファイル: components/NotificationBell.tsx)
   - アクション: NotificationBellとNotificationListモーダルを作成し、Supabaseリアルタイム購読を設定
   - なぜ: ユーザーがリアルタイムで通知を受信・確認できるようにするため
   - 依存関係: フェーズ1完了
   - リスク: 低

## テスト戦略
- ユニットテスト: lib/notifications.ts（キュー処理、再試行ロジック）
- 統合テスト: マーケット解決 → 通知エンキュー → 配信フロー
- E2Eテスト: ユーザーがマーケット解決通知を受信するフロー

## リスクと軽減策
- **リスク**: メール到達率の確保
  - 軽減策: SPF/DKIM設定の事前確認
- **リスク**: 大規模マーケット（1000人以上）でのパフォーマンス
  - 軽減策: バッチ処理と非同期キューで対応

## 成功基準
- [ ] マーケット解決時に対象ユーザーへ通知が送信される
- [ ] 配信失敗時に再試行が実行される
- [ ] 通知設定の変更が反映される

**確認待ち**: この計画で進めますか？（yes / はい / modify / 修正）
```

## 重要な注意事項

**重要**: plannerエージェントは、「yes」「はい」「proceed」または類似の肯定的な応答で計画を明示的に確認するまで、コードを書きません。

変更が必要な場合は、以下のように応答してください:

- "modify: [変更内容]" / "修正: [変更内容]"
- "別のアプローチ: [代替案]"
- "フェーズ2をスキップしてフェーズ3を先に実行"

## 他のコマンドとの統合

- ビルドエラーが発生した場合は`/build-fix`を使用
- 完了した実装をレビューするには`/code-review`を使用
